# Ralph Progress

## Codebase Patterns
- `chatlog_alpha` is a Go local-tool product for Chatlog CLI/TUI/HTTP, WeChat local database access, semantic search, temporal graph processing, and daily reports.
- Start every Ralph iteration by reading root `AGENTS.md`, `skills/chatlog-http-cli/SKILL.md`, `feature_list.json`, root `progress.md`, root `session-handoff.md`, `scripts/ralph/prd.json`, and this file.
- Use `./init.sh` as the default quick gate; use focused `go test ./internal/...` for Go logic; use `./init.sh --full` only for broad repo changes.
- Do not run `chatlog report daily --vision` or `chatlog report daily --summary` unless the PRD story and user authorization explicitly require model/vision calls.
- Verify private chat-derived workflows through paths, timestamps, counts, HTTP status, structured fields, or error buckets; do not print private message or report contents.
- Developer agents do not commit directly. `scripts/ralph/ralph.py` commits only after Validator success and excludes startup-dirty paths plus private/cache output directories.
- Long runtime windows (>2min) should run via `nohup script.sh &` (or a TaskCreate with `run_in_background`); the parent shell returns immediately, the child keeps sampling. Snapshot to `scripts/ralph/progress.txt` not a new file, and judge PASS/blocked only after the loop finishes, never mid-loop.
- `prd.json` field is `userStories`, not `stories`. After any PRD edit run `jq '.userStories | length'` to confirm shape.
- For MiniMax key pool: stable ordinal labels (`key_1`, `key_2`, ...) are preferred over `***xxxx` tail-redaction because operators want to grep for "which key hit the wall". A non-retryable bucket (`minimax_sensitive_1026/1027`, `minimax_decode_error`, ...) must NEVER move a key into pool-level quarantine — only `minimax_auth_error` and the model-retryable buckets the operator has explicitly opted into should. Test the negative case (`TestRecordErrorDoesNotQuarantineOnSensitiveOrRateOrDecode`) so future refactors cannot silently rotate 1026/1027 keys.
- `chatMMXRaw.recordError` is only called after a successful `Acquire` (i.e. `ensureLocked` already ran), so recordError can rely on `p.slots` being populated. Tests that call `recordError` before any `Acquire` must touch Acquire first to materialize slots.
- HTTP handler 暴露 snapshot 字段的最佳实践: 当 snapshot 结构已包含目标字段(类型/JSON tag 完备)但 HTTP gin.H 没 spread 时,**只需在 handler 的 gin.H literal 里把缺失字段逐项写出**即可一行修复,不必重构 snapshot 端或加 spread helper。这样修改面最小、影响面最小,其它调用 snapshot 的代码不受牵连。修复后用 dummy key + 真实 endpoint 触发一次 `recordError` 即可在 3 秒内观察到 `quarantined_key_count=5`。
- `passes=true` + 非空 `notes` 会让 `ralph.py:story_ready_to_commit()` 沉默跳过自动 commit。修"重试"类 story 必须三步走:`passes=true` + `notes=""` + `retryCount=0`,否则 ralph 自动 commit 会被沉默跳过,导致 commit 链路断裂。
- 自适应 worker cap (HA-003) 状态机设计: 3 个 tier (stable=5 / degraded=3 / critical=1),`windowDuration=60s` + `recoveryGrace=90s` 故意让 grace > 窗口,这样窗口清空后还有 grace 缓冲,避免 spike 簇在窗口边缘震荡;downgrade 立即生效,upgrade 必须 `previous != stable && QuietDuration >= recoveryGrace`。spike bucket 严格只接受 `minimax_rate_limited / minimax_timeout / minimax_before_request_timeout` — `minimax_sensitive_1026/1027 / minimax_decode_error / minimax_auth_error / minimax_config_error` 显式不参与(让 `tracker.Observe` 过滤,不要让 manager 在上层分支判断),保证内容安全/解析问题不被 worker 降并发掩盖。
- 自适应 worker cap 的"事实"边界: `adaptiveMaxWorkers` 必须把上层 `min(configured_key_count, maxGraphWorkers, userSetWorkers)` 当作硬上限,自己只往下降,绝不越权给用户/运维放大到他们没设的 worker 数;`ProcessPending` 每次 batch 都重算 `effectiveWorkers()`,Manager 缓存 `lastEffectiveWorkers` 给 `Status()` 用,让 `/api/v1/graph/status` 不需要重算。
- HA-004 failed-source bucket 分类器的"先归一再决策"原则: 不要在 `requeueRecoverableFailedSources` 里散落 if-else 判断每个非可恢复桶,把所有错误归一进一个**稳定 10 类 bucket 词表**(`config_error` / `network_timeout` / `before_request_timeout` / `rate_limited` / `auth_error` / `json_decode_error` / `empty_graph` / `sensitive_input_1026` / `sensitive_output_1027` / `non_retryable_request` + `unclassified`),requeue 路径只对**可恢复桶名**的闭合列表(`recoverableGraphBuckets`, 4 个)逐个调 `RequeueFailedSourcesByBucket`,manager 完全不需要知道每个 token 长什么样。这样后续加新桶(比如 `minimax_internal_error`)只需加分类器 token + 名字,不动 requeue 决策;且 `ClassifyFailedError` 是 substring match,顺序决定优先级,recoverable 桶放前面,non-retryable 放后面,避免重叠 token 误判。
- 桶计数 `FailedBucketCounts()` 必须返回**全 10 个桶 + unclassified** 的 map 即便 count=0,这样 `/api/v1/graph/status#failed_buckets` 永远稳定暴露所有键,HA guard 端不用做 nil-check;同时 Status 暴露的 buckets 只含 name+count,不含 source payload / prompt / 私聊正文,符合隐私安全。
- 测试桶分类器时**正反两路都要 lock**: `TestClassifyFailedErrorCoversAllHABuckets` 锁 10 类 positive + 2 类 negative (empty / unknown),`TestAllFailedBucketNamesIsStableAndComplete` 锁 list 顺序,`TestManagerRequeueRecoverableOnlyMovesRecoverableBuckets` 端到端验证 manager 只 requeue 4 类、保留 6 类,只要未来有人改 `recoverableGraphBuckets` 或 token 列表,测试会立刻显红,避免 silent rotation 1026/1027 进 pending。
- 测试"调用外部服务"的代码:首选 **接口注入 + 函数适配器**(`type ChatInvoker interface { Chat(...) (string, error) }` + `type ChatInvokerFunc func(...) (string, error)`),fake 实现就是 plain Go function,生产路径用 `ChatInvokerFunc(client.Chat)` 一行包装,既不需要 reflect/mock 框架,也不需要在 `*Client` 上加 hook;benchmark 类的代码必须能离线单测,dry-run 默认 `--execute` opt-in 是隐私安全的硬护栏。
- Benchmark/gate 边界测试模式: 把 gate 阈值(`JSONValidRateGap=0.02` / `NonEmptyGraphRateGap=0.05`)显式命名常量 + 写 4 个边界测试(`PassWhenMeets` / `RejectOnJSONValidRegression` / `RejectOnNonEmptyRegression` / `AtBoundaryAllows` / `SlightlyPastBoundaryRejects`)。任何给小数位后两位的微调都不会让测试假绿/假红,review 时只看 const 定义就能对上线门禁有"绝对值"的感觉。
- Fixture 隐私 + 覆盖的硬护栏模式: 任何 "测试 fixture" 必须 (1) 用 const 显式 allow-list 7 类分类(短句/省略语/时间/多人/业务/关系/空) + `TestXxxCoversAllRequiredCategories` 锁"7 类都覆盖",(2) `TestXxxHasNoRealChatContent` 用反例 `wxid_` / `13800` grep 锁住"不能塞真实聊天",(3) `TestXxxHasVariedShape` 锁"每类至少 1 个 fixture"。这样 review 想删一类 / 塞私货都会立刻显红,privacy 是测试 invariant 不是文档声明。
- HA guard productize 的 "dry-run 不真做 + 实时 SUMMARY 一行化" 模式: 把所有 mutating helper (`restore_semantic_config` / `restart_service` / `ensure_graph_config` / `resume_graph_if_needed`) 改成只 `plan_action "<name> <key=val...>"` 收口到一个 `PLANNED_ACTIONS[]` 数组,`run_or_print` 仍然能切换真/假执行;末尾 `emit_summary_block` 把 `GUARD_HEALTH / SEMANTIC_PROVIDER / MMX_CONFIGURED / GRAPH_WORKERS / GRAPH_PENDING / GRAPH_FAILED / action_taken / planned_count` 用 key=value 拼成一行;`--loop 60` 直接复用 `check_once` + 末尾加 SUMMARY tick 行。这样 AC#1 (完整 dry-run 摘要) 和 AC#4 (loop 每轮时间戳+状态+动作) 一次性满足,且 AC#2 (服务健康时 planned_count=0 → 真正幂等) 只需要保证 helper 的 plan_action 调用前置判断准。
- 进程级健康检查 (`check_service_process` 模式): guard 不止看 HTTP `/health`,还要用 `lsof -nP -iTCP:5030 -sTCP:LISTEN -t` 拿 listener PID,`ps -p PID -ww -o command=` 取完整 cmdline,`ps eww -p PID` 取 env blob,这样才能判 (a) 是否 `serve --config-dir <dir>`、(b) `MINIMAX_API_KEYS` 数量。AC#3 的"进程不是规范启动" 必须用 `lsof` + `ps` 走真实 OS 层,不能只靠 HTTP status;HTTP 全 ok 但 cmdline 漂移是 2026-06-07 实际 production 遇到的故障模式,guard 应当捕获并 respawn。
- macOS `mv` 在 immutable flag 文件上会报 `set owner/group` 警告但写入成功;用 `jq > tmp && mv tmp target` 三步走也是同样警告。验证时只 check `jq` 输出,不要被警告吓到。
- `notes` 非空 → ralph.py 沉默跳过自动 commit。修复类 story 的 "三步收口" 标准操作: `passes=true` + `notes=""` + `retryCount=0`。
- HA-007 E2E 收口:5 个 AC 拆到 status 字段级别,每个对应 1 个 HTTP 端点 + 1 个可重复的取值/动作;recoverable/non-recoverable 共存性用 "一次 resume 同时验证 pending/processed 动 + sensitive_1026/1027 不动" 一击两证;binary 字段缺失时用 `sqlite3 -readonly` 按 HA-004 桶分类正则派生等价证据(SQLite 永远是 ground truth,比 HTTP 字段更鲁棒);长窗口用 `nohup /tmp/ha007_sample.sh &` + pidfile 等退出模式(父 shell `while [ -f pidfile]; do sleep 5; done`),不阻塞主流程。

- **未来迭代的学习：**
  - 可复用 patterns
    - 桶分类器的"先归一再决策"是反散落分支的通用模板: 任何"X 条件下做 A,Y 条件下做 B"类恢复逻辑都应该先把条件归一到稳定词表(10 个),决策端只对**桶名**的闭合列表遍历,不要在决策端散落 substring 匹配。这样加新桶不动决策、未来有人提 PR 改 requeue 列表,review 是一行 `[]string{...}` 而非 50 行 token 拼接。
    - 桶计数 API 必须返回**全字段**的 map 即便 count=0,HTTP caller / HA guard 不需要做 nil-check,前端 dashboard 也不需要补 "missing bucket" 占位。`AllFailedBucketNames` + `FailedBucketCounts` 这种"先有名字,后填值"的两步设计是 API 稳定性的核心。

    - `ClassifyFailedError` 用 substring match 时,recoverable 桶 token 必须在 non-recoverable 桶 token 之前(顺序敏感),否则一个含多个 token 的错误会被错分到 non-retryable 类。本轮 10 类顺序是 `config_error` → `network_timeout` → `before_request_timeout` → `rate_limited` → `auth_error` → `json_decode_error` → `empty_graph` → `sensitive_input_1026` → `sensitive_output_1027` → `non_retryable_request`,前 4 个是 recoverable,后 6 个是 non-recoverable,顺序就是"先可恢复再不可恢复"避免误判。
    - `RequeueFailedSourcesByBucket` 不能直接把 caller 传的字符串当 SQL fragment 拼,必须用 `failedBucketTokens[bucket]` 查词表取 token 列表后再 LIKE `%token%`;未知 bucket 名 no-op(返 0)而不是返错,这样 caller 可以安全传 `m.store.RequeueFailedSourcesByBucket(req.Bucket)` 而不踩 SQL injection 风险。
    - `Status.FailedBuckets` 字段加了 `omitempty` 但因为 `map[string]int` 零值是 nil,`omitempty` 在 nil 时会省略字段 — 如果未来某次 counts 读取失败,JSON 会缺 `failed_buckets` 键而不是返空 map。HA guard 端需要容忍 "key 不存在" 和 "key 存在但全 0" 两种情况。本轮没改这行为,但记下来供 HA-006 review 时一并考虑。

- **未来迭代的学习:**
  - 可复用 patterns
    - "E2E AC 拆解到 status 字段级别" 的 HA-007 收口:5 个 AC 每一个都对应 1 个 HTTP 端点 + 1 个可重复的取值/动作,validator 跑完一遍 = 5 条 curl 就能复现,不需要跑完整 suite。这是 E2E story 设计的正确分解,AC 写得越具体,validator 越快。
    - "recoverable 行为共存" 验证模式: AC#3 同时检查 (a) pending/processed 变化(recoverable 在动)和 (b) sensitive_1026/1027 不动(non-recoverable 不动),一次 resume 调用同时证伪/证实两个不变性,比拆两个独立 AC 节省时间且更有说服力。
    - "SQLite readonly 兜底" 模式: 当生产 binary 还没含新字段(HA-004 failed_buckets),用 `sqlite3 -readonly <db> "SELECT CASE WHEN error LIKE '...' THEN bucket ..."` 派生等价证据。SQLite 永远是 ground truth,比依赖 HTTP 字段更鲁棒。
    - "10min × 60s nohup loop + 等待 pidfile 退出" 模式: `nohup /tmp/ha007_sample.sh &` 起后台,记录 PID 到 pidfile,父 shell `while [ -f pidfile ]; do sleep 5; done` 等退出。比 `TaskCreate` + 轮询更轻量,比 `sleep 600` 不阻塞父 shell 上下文。
  - 遇到的陷阱
    - `prd.json` 里的 `failed_buckets` 字段在生产 binary 不可见 — HA-004 加的字段未编译,AC#3 的"敏感不被 requeue"证据必须用 SQLite 兜底。**下次 HA-x 收口时,先看 binary timestamp vs 字段引入时间,决定走 HTTP 字段还是 SQLite 兜底**。
    - `nohup /tmp/ha007_sample.sh &` 在 zsh 下的 `$$` 解析正确,但在嵌套 subshell 里 `$$` 可能取父 PID,需要用 `$BASHPID`(bash)或在子 shell 顶部 `echo $$ > pidfile` 显式落盘。本轮脚本在子 shell 顶部写 pidfile,无问题。
    - macOS `mv` 在 immutable flag 文件上 `set owner/group (was: 501/0): Operation not permitted` 警告但写入成功(已用 `jq` 二次确认目标文件内容正确),不要被警告吓到。Codebase Patterns 已有此条目。
