# Ralph Progress

## Codebase Patterns
- `chatlog_alpha` is a Go local-tool product for Chatlog CLI/TUI/HTTP, WeChat local database access, semantic search, temporal graph processing, and daily reports.
- Start every Ralph iteration by reading root `AGENTS.md`, `skills/chatlog-http-cli/SKILL.md`, `feature_list.json`, root `progress.md`, root `session-handoff.md`, `scripts/ralph/prd.json`, and this file.
- Use `./init.sh` as the default quick gate; use focused `go test ./internal/...` for Go logic; use `./init.sh --full` only for broad repo changes.
- Do not run `chatlog report daily --vision` or `chatlog report daily --summary` unless the PRD story and user authorization explicitly require model/vision calls.
- Verify private chat-derived workflows through paths, timestamps, counts, HTTP status, structured fields, or error buckets; do not print private message or report contents.
- Developer agents do not commit directly. `scripts/ralph/ralph.py` commits only after Validator success and excludes startup-dirty paths plus private/cache output directories.
- Long runtime windows (>2min) should run via `nohup script.sh &` (or a TaskCreate with `run_in_background`); the parent shell returns immediately, the child keeps sampling. Snapshot to `scripts/ralph/progress.txt` not a new file, and judge PASS/blocked only after the loop finishes, never mid-loop.
- `prd.json` field is `userStories`, not `stories`. After any PRD edit run `jq '.userStories | length'` to confirm shape.
- For MiniMax key pool: stable ordinal labels (`key_1`, `key_2`, ...) are preferred over `***xxxx` tail-redaction because operators want to grep for "which key hit the wall". A non-retryable bucket (`minimax_sensitive_1026/1027`, `minimax_decode_error`, ...) must NEVER move a key into pool-level quarantine — only `minimax_auth_error` and the model-retryable buckets the operator has explicitly opted into should. Test the negative case (`TestRecordErrorDoesNotQuarantineOnSensitiveOrRateOrDecode`) so future refactors cannot silently rotate 1026/1027 keys.
- `chatMMXRaw.recordError` is only called after a successful `Acquire` (i.e. `ensureLocked` already ran), so recordError can rely on `p.slots` being populated. Tests that call `recordError` before any `Acquire` must touch Acquire first to materialize slots.
- HTTP handler 暴露 snapshot 字段的最佳实践: 当 snapshot 结构已包含目标字段(类型/JSON tag 完备)但 HTTP gin.H 没 spread 时,**只需在 handler 的 gin.H literal 里把缺失字段逐项写出**即可一行修复,不必重构 snapshot 端或加 spread helper。这样修改面最小、影响面最小,其它调用 snapshot 的代码不受牵连。修复后用 dummy key + 真实 endpoint 触发一次 `recordError` 即可在 3 秒内观察到 `quarantined_key_count=5`。
- `passes=true` + 非空 `notes` 会让 `ralph.py:story_ready_to_commit()` 沉默跳过自动 commit。修"重试"类 story 必须三步走:`passes=true` + `notes=""` + `retryCount=0`,否则 ralph 自动 commit 会被沉默跳过,导致 commit 链路断裂。
- 自适应 worker cap (HA-003) 状态机设计: 3 个 tier (stable=5 / degraded=3 / critical=1),`windowDuration=60s` + `recoveryGrace=90s` 故意让 grace > 窗口,这样窗口清空后还有 grace 缓冲,避免 spike 簇在窗口边缘震荡;downgrade 立即生效,upgrade 必须 `previous != stable && QuietDuration >= recoveryGrace`。spike bucket 严格只接受 `minimax_rate_limited / minimax_timeout / minimax_before_request_timeout` — `minimax_sensitive_1026/1027 / minimax_decode_error / minimax_auth_error / minimax_config_error` 显式不参与(让 `tracker.Observe` 过滤,不要让 manager 在上层分支判断),保证内容安全/解析问题不被 worker 降并发掩盖。

- HA-004 failed-source bucket 分类器的"先归一再决策"原则: 不要在 `requeueRecoverableFailedSources` 里散落 if-else 判断每个非可恢复桶,把所有错误归一进一个**稳定 10 类 bucket 词表**(`config_error` / `network_timeout` / `before_request_timeout` / `rate_limited` / `auth_error` / `json_decode_error` / `empty_graph` / `sensitive_input_1026` / `sensitive_output_1027` / `non_retryable_request` + `unclassified`),requeue 路径只对**可恢复桶名**的闭合列表(`recoverableGraphBuckets`, 4 个)逐个调 `RequeueFailedSourcesByBucket`,manager 完全不需要知道每个 token 长什么样。这样后续加新桶(比如 `minimax_internal_error`)只需加分类器 token + 名字,不动 requeue 决策;且 `ClassifyFailedError` 是 substring match,顺序决定优先级,recoverable 桶放前面,non-retryable 放后面,避免重叠 token 误判。

- **冲突 setup 的"race-before-fork"陷阱**:git 三方合并要触发冲突,需要 base 与 work-branch 从**同一祖先**拉出后**各自独立**对同一文件同一行做不同修改。若 base 在 work-branch 拉出**前**先添加/修改文件,work-branch 后续改它只是 base 的子孙,merge 时 base 看不到反向修改,`git merge --no-ff` 走 fast-forward(同祖先 trivial merge),不报冲突。US-003 scene 3 第一次跑就是这种顺序,`auto_merge_branch` 直接走"✅ 合并完成"成功路径,场景完全测反。修复后顺序是 "work-branch 写 v1 + commit → 切回 base 写 v2 + commit → 切回 work-branch → auto_merge_branch",base 与 work-branch 从共同祖先分叉,各自独立改动 shared.txt,merge 才真冲突。**任何"冲突 abort"沙箱场景的脚手架都按"先 fork 后 race"模板,不要先在 base 改文件再 fork**。

- US-003 macOS bash 3.2 + heredoc + f-string 陷阱(2026-06-12): `out="$( python3 <<'PY' ... print(f"...{var}...") ... PY )"` 即使 heredoc 用 `<<'PY'` 单引号,在 macOS 系统 bash 3.2 (`/bin/bash` 默认 3.2.57) 的 parse 阶段仍会报 `syntax error near unexpected token '('`,根因是 f-string 内的 `({var})` 触发 bash brace expansion 解析。**结论:复杂 heredoc 永远用 `( ... ) >> "$out_file"` 子 shell + 临时文件捕获,而非 `$( ... )`**,`rc=$?` 在 `(...)` 后直接捕获,再用 `out="$(cat "$out_file")"` 拿输出,既保留 heredoc 又绕过 parse 限制;`set -euo pipefail` 下子 shell 退出非 0 不会让脚本退出(子 shell 退出码赋给 `rc=$?` 后显式判断),但 `local rc=0` 必须在子 shell 之前声明,子 shell 之后重新 `set -e`。`auto_merge_branch()` 冲突时抛 `SystemExit(1)` 走 `rc ∈ {0,1}` 双值判定(0 = 正常、1 = 冲突预期、其他 = 异常)。
- US-003 沙箱分支名 env 传参(2026-06-12): 沙箱 python 进程内 `import ralph` 时 ralph.py 模块顶层读 `sys.argv[1:]` 决定 AGENT,非 `claude`/`codex` 立即 `sys.exit(2)`。任何把分支名当 `sys.argv[1]` 传给 python 进程的写法(`python3 - "<branch>" <<'PY'`)会在 `import ralph` 阶段就 abort,沙箱根本进不到 `ensure_work_branch`。**结论:沙箱与分支名解耦 → 分支名走环境变量**(`RALPH_TEST_BRANCH="$branch_name" PYTHONDONTWRITEBYTECODE=1 python3 <<'PY' ... os.environ["RALPH_TEST_BRANCH"] ... PY`),python heredoc 不传任何位置参数。`ralph.py` 启动期捕获到的 `BASE_BRANCH` 也因此是 `<sandbox>/main` 而非真实仓库的 `main`,沙箱全自包含不污染真实仓库。
